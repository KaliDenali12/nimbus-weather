## 1. Architecture Overview

Nimbus is a **pure client-side** application. It does not expose any HTTP endpoints. All API interaction is outbound â€” the app consumes two Open-Meteo REST endpoints via the browser `fetch` API.

All HTTP calls are centralized in a single module: `src/lib/api.ts`.

### External API Endpoints Consumed

| Provider | Endpoint | Purpose |
|----------|----------|---------|
| Open-Meteo | `GET https://geocoding-api.open-meteo.com/v1/search` | City name search (geocoding) |
| Open-Meteo | `GET https://api.open-meteo.com/v1/forecast` | Current weather + daily forecast |

### Internal API Functions

| Function | File | Throws? | Returns |
|----------|------|---------|---------|
| `searchCities(query)` | `api.ts` | Yes (`ApiError`) | `GeocodingResult[]` |
| `fetchWeather(lat, lon, name, country)` | `api.ts` | Yes (`ApiError`) | `WeatherData` |
| `reverseGeocode(lat, lon)` | `api.ts` | No (returns `null`) | `{name, country} \| null` |

## 2. URL Construction

### Convention: Use `URL` + `searchParams`

Always construct URLs using the `URL` constructor and `searchParams.set()`. Never concatenate query strings manually.

```typescript
// GOOD
const url = new URL(BASE_URL)
url.searchParams.set('name', query.trim())
url.searchParams.set('count', '8')

// BAD
const url = `${BASE_URL}?name=${query}&count=8`
```

### Base URL Constants

Define base URLs as module-level constants with `UPPER_SNAKE_CASE`:

```typescript
const GEOCODING_URL = 'https://geocoding-api.open-meteo.com/v1/search'
const FORECAST_URL = 'https://api.open-meteo.com/v1/forecast'
```

## 3. Field Naming Conventions

### External API fields (Open-Meteo): `snake_case`
The Open-Meteo API uses `snake_case` for all fields (`temperature_2m`, `wind_speed_10m`, `country_code`).

### Internal app types: `camelCase`
All internal TypeScript interfaces use `camelCase` for field names.

### Mapping Convention

The API module (`api.ts`) is the **single boundary** where `snake_case` â†’ `camelCase` transformation occurs. No other module should reference Open-Meteo field names.

| Open-Meteo Field | Internal Field | Type |
|------------------|----------------|------|
| `temperature_2m` | `temperature` | `number` |
| `apparent_temperature` | `feelsLike` | `number` |
| `relative_humidity_2m` | `humidity` | `number` |
| `wind_speed_10m` | `windSpeed` | `number` |
| `weather_code` | `weatherCode` | `number` |
| `is_day` | `isDay` | `boolean` (converted from 0\|1) |
| `temperature_2m_max` | `tempHigh` | `number` |
| `temperature_2m_min` | `tempLow` | `number` |
| `precipitation_probability_max` | `precipitationProbability` | `number` |

### Internal Field Patterns

- **Booleans**: `camelCase` with `is` prefix for state flags (`isDay`, `isDarkMode`)
- **IDs**: plain `id` (number)
- **Coordinates**: `latitude` / `longitude` in types, `lat` / `lon` in shorthand (`City` type)
- **Timestamps**: ISO string format stored as `string` (e.g., `date: string`)

## 4. Error Handling

### Error Class

Use the custom `ApiError` class for all HTTP-level errors:

```typescript
export class ApiError extends Error {
  status?: number
  constructor(message: string, status?: number) {
    super(message)
    this.name = 'ApiError'
    this.status = status
  }
}
```

### Error Strategy by Function Type

| Strategy | When to Use | Example |
|----------|-------------|---------|
| **Throw `ApiError`** | Primary data-fetching functions where failure must be handled by the caller | `searchCities`, `fetchWeather` |
| **Return `null`** | Auxiliary/best-effort functions where failure is acceptable | `reverseGeocode` |

### Error Messages

- Use human-readable, action-oriented messages: `"Failed to search cities"`, `"Failed to fetch weather data"`
- Include the HTTP status code as `ApiError.status` for debugging
- Never expose raw API error bodies, stack traces, or internal paths to the user

### Context Layer Error Handling

The `WeatherContext` catches all `ApiError` throws and stores the message in `error` state. Components display this via `ErrorState` with a "Try Again" button.

## 5. Input Validation

### Convention: Validate at the API Function Boundary

Each API function validates its inputs before making a network request:

```typescript
// GOOD: Validate before fetch
export async function searchCities(query: string): Promise<GeocodingResult[]> {
  if (query.trim().length < 2) return []  // No-op for short queries
  // ... fetch
}
```

### Validation Rules

| Function | Validation | Behavior |
|----------|-----------|----------|
| `searchCities` | `query.trim().length < 2` | Returns `[]` (no fetch) |
| `fetchWeather` | None (trusts caller) | Delegates to Open-Meteo |
| `reverseGeocode` | None (trusts caller) | Rounds coords, returns `null` on failure |

## 6. Response Transformation

### Convention: Transform at the API Boundary

The API module transforms external response shapes into internal TypeScript interfaces. No other module should parse raw API data.

### Default Values

When Open-Meteo omits optional fields, use sensible defaults:

| Field | Default | Rationale |
|-------|---------|-----------|
| `precipitation_probability_max` | `0` | Missing = no precipitation data |
| `timezone` | `'UTC'` | Safe universal fallback |
| `country` (reverse geocode) | `''` | Empty string over `undefined` |
| `alerts` | `[]` | Free tier limitation â€” always empty |

### Type Coercions

| Source | Target | Rule |
|--------|--------|------|
| `is_day: 0 \| 1` | `isDay: boolean` | `=== 1` comparison |
| Column arrays (daily) | Row objects (`DailyForecast[]`) | `.map()` over `time` array |

## 7. Data Persistence (localStorage)

### Key: `nimbus-preferences`

```typescript
interface UserPreferences {
  unitPreference: 'celsius' | 'fahrenheit'
  darkModeEnabled: boolean
  recentCities: City[]  // max 5, FIFO with lat/lon dedup
}
```

### Conventions

- Always wrap `localStorage` access in try/catch (may be full or unavailable)
- Return defaults on any parse error (never throw from storage)
- Limit collection sizes (`MAX_RECENT_CITIES = 5`)
- Deduplicate by coordinates (`lat` + `lon`), not by name

## 8. Testing API Code

### Unit Tests (per function)

- Mock `globalThis.fetch` â€” never make real network calls
- Test happy path, empty results, error responses, and missing fields
- Verify URL construction (correct base URL, query params)

### Contract Tests

- Validate response objects conform to TypeScript interfaces
- Test field mappings from external â†’ internal naming
- Test type coercions (integer â†’ boolean, column â†’ row)
- Test default values for missing/optional fields

### Integration Tests

- Test full data flow: geolocation â†’ search â†’ fetch â†’ context â†’ component
- Test fallback behavior (Antarctica, retry)
- Test preference persistence

## 9. Adding New API Calls

When adding a new external API call, follow this checklist:

1. Add the base URL constant in `api.ts`
2. Define request/response types in `types/weather.ts`
3. Create the async function in `api.ts`
4. Use `URL` + `searchParams` for URL construction
5. Transform response to internal types at the API boundary
6. Decide error strategy: throw `ApiError` (primary) vs return `null` (auxiliary)
7. Add input validation before the fetch call
8. Write unit tests + contract tests in `lib/__tests__/`
9. Integrate via `WeatherContext` if the data feeds the UI
